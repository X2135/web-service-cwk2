# 深度 GenAI 评估报告（批判性草稿）

## 1. 使用范围

本项目在以下环节使用了 GenAI 辅助：
- 项目结构与模块划分建议
- 测试用例设计思路
- 文档草稿、基准脚本与 CI 配置建议
- 排序算法优化思路（TF‑IDF、BM25）

AI 的角色是“辅助设计与加速实现”，不是最终的代码审阅者或架构决策者。

## 2. 带来的实际收益

### 2.1 提高开发速度
AI 能快速给出脚手架、文档模板与常见实现模式，减少从零开始搭建的时间。

### 2.2 扩展方案空间
对于搜索引擎这种包含爬虫、索引、排序和 CLI 的多模块系统，AI 可以同时提供多个方向的候选方案，帮助快速比较取舍。

### 2.3 促进测试意识
AI 往往会主动建议补充测试、边界条件和错误处理，这对课程项目很有帮助。

## 3. 主要局限与风险

### 3.1 代码正确性并不自动成立
AI 生成的代码可能在边界条件、导入路径、缓存一致性或性能假设上出错。本项目中就出现过：
- 测试依赖未正确设置 `PYTHONPATH`
- 合成 benchmark 的词表不满足索引器正则规则
- BM25 / TF‑IDF 结果缓存键需要显式包含排序方法

这些问题都说明：AI 的输出必须经过人工运行、验证和修正。

### 3.2 容易产生“看起来合理”的错误
AI 有时会给出语义正确但工程上不完整的建议，例如：
- 只讨论功能，不考虑可复现性
- 给出性能建议但没有可执行基准
- 写出文档结构，却遗漏实际命令或依赖说明

### 3.3 可能放大偏见或过度自信
如果不加审查，AI 可能把“推荐做法”包装成“唯一正确做法”。实际工程应根据需求、评分标准和约束条件选择，而不是盲目照搬。

## 4. 人工控制与验证措施

本项目对 AI 辅助结果采取了以下控制：
- 每次改动后运行单元测试或性能检查
- 使用 mock 隔离网络和外部依赖
- 为关键排序逻辑增加可重复的测试
- 把 benchmark 结果写入文件，避免仅依赖口头描述
- 通过 README、架构文档和评分对照表把实现和评分要求对应起来

## 5. 伦理与学术诚信

在课程项目中，使用 AI 辅助应满足以下原则：
- 最终提交内容必须能被作者解释与维护
- 不应提交无法理解或无法复现的 AI 代码
- 需要明确哪些部分由 AI 辅助完成，哪些部分由人工验证
- 对外部数据抓取要注意网站条款、礼貌抓取和合法性

## 6. 结论

GenAI 在本项目中是有效的生产力工具，尤其适合加速：
- 模块化设计
- 测试补全
- 文档整理
- 基准与 CI 搭建

但它不能替代：
- 需求判断
- 代码审查
- 真实运行验证
- 对评分标准的逐项对齐

因此，最佳实践是“AI 提速 + 人工把关 + 自动化验证”。
